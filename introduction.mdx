# Welcome to NduloJS

An opinionated TypeScript backend framework built on [Elysia](https://elysiajs.com) that enforces Clean Architecture structurally — dependency injection, explicit error handling, and layered modules — so your codebase stays consistent as it grows.

<Note>
  NduloJS is designed to help you build scalable backend applications with enforced best practices from day one.
</Note>

## Why NduloJS?

NduloJS eliminates architectural debates by providing a structured, opinionated approach to backend development. No more wondering how to organize your code, handle errors, or manage dependencies — it's all built in.

## Key features

### Type-safe error handling

Every handler returns `Ok(value)` or `Err(error)`. The framework automatically maps errors to HTTP status codes:

```typescript
app.get('/users/:id', async ({ params }) => {
  const user = await db.findById(params.id);
  if (!user) return Err(ErrorFactory.notFound('User not found', 'User', params.id));
  return Ok(user);
});
```

<Tip>
  The Result pattern eliminates uncontrolled exceptions and makes error handling explicit in your type signatures.
</Tip>

### Functional dependency injection

No decorators, no metadata reflection — just pure functions:

```typescript
const container = createContainer()
  .register('Config',         ()  => loadConfig())
  .register('Database',       (c) => createDatabase(c.resolve('Config')))
  .register('UserRepository', (c) => createUserRepository(c.resolve('Database')))
  .register('UserService',    (c) => createUserService(c.resolve('UserRepository')));

const userService = container.resolve('UserService');
```

### Enforced Clean Architecture

Generate modules with the CLI and get proper layering automatically:

```bash
ndulo generate module farms
```

Creates:

```
src/modules/farms/
├── events/farm.events.ts
├── application/
│   ├── dtos/farm.dto.ts
│   ├── ports/farm.port.ts
│   └── services/farm.service.ts
└── infrastructure/
    ├── persistence/farm.repository.ts
    └── http/controllers/farm.controller.ts
```

## Error types mapping

All errors are automatically mapped to appropriate HTTP status codes:

| Error Factory Method | HTTP Status | Use Case |
|---------------------|-------------|----------|
| `ErrorFactory.notFound()` | 404 | Resource doesn't exist |
| `ErrorFactory.unauthorized()` | 401 | Missing or invalid authentication |
| `ErrorFactory.forbidden()` | 403 | Authenticated but not allowed |
| `ErrorFactory.validation()` | 422 | Input validation failed |
| `ErrorFactory.conflict()` | 409 | Resource already exists |
| `ErrorFactory.business()` | 400 | Business rule violation |
| `ErrorFactory.database()` | 500 | Database operation failed |
| `ErrorFactory.externalService()` | 502 | External API call failed |
| `ErrorFactory.tooManyRequests()` | 429 | Rate limit exceeded |

<Warning>
  Always use `ErrorFactory` to create errors. This ensures proper error structure and automatic HTTP status mapping.
</Warning>
